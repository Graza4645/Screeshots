## Project Overview

This is a **Playwright POM (Page Object Model) Automation Framework** built using Kiro AI. It includes:

- **Page Object Model** pattern for maintainable test code
- **Two test modules:** Login and Salary Calculator
- **MCP integrations:** Playwright (browser inspection) and Jira (ticket-based test generation)
- **Auto-heal system:** Kiro hook that automatically detects test failures, diagnoses root cause, fixes code, and re-runs failed tests
- **Built-in retry:** Playwright retries failed tests once locally, twice in CI
- **`--last-failed` support:** Re-run only previously failed tests

### Auto-Heal Hook

**Path:** `.kiro/hooks/auto-heal-tests.kiro.hook`

```json
{
  "enabled": true,
  "name": "Auto-Heal Failing Tests",
  "description": "After the agent stops, checks if the last action was a test run that failed. If so, automatically analyzes the failure, fixes the root cause (locators, test data, page object methods, or assertions), and re-runs the failing test to confirm the fix.",
  "version": "1",
  "when": {
    "type": "agentStop"
  },
  "then": {
    "type": "askAgent",
    "prompt": "Check if the last action involved running Playwright tests (npx playwright test). If tests FAILED:\n\n1. Read the test failure output carefully — identify which test(s) failed and the exact error message.\n2. Diagnose the root cause. Common causes:\n   - Locator not matching (element changed on page) → Use MCP Playwright browser_navigate + browser_snapshot to inspect the actual page, then update the locator in the PageObject file.\n   - Timeout waiting for element → Check if the page loaded correctly, add proper waits.\n   - Assertion mismatch → Check expected vs actual values, update test data or assertion.\n   - Import path error → Verify relative paths follow the ../PageObject/ and ../testData/ convention.\n   - Navigation failure → Verify the URL in testData is correct and reachable.\n3. Fix the identified issue in the correct file (PageObject, Script, or testData).\n4. Re-run ONLY the failed tests using: cmd /c \"npx playwright test --last-failed\"\n   This uses Playwright's built-in --last-failed flag which automatically picks up only the tests that failed in the previous run.\n5. If all pass, report success. If tests fail again with a DIFFERENT error, repeat the diagnosis. If the same error persists after 2 fix attempts, stop and report the issue to the user.\n\nIMPORTANT: Do NOT re-run if tests already PASSED. Only trigger healing on failures."
  }
}
```

**How auto-heal works:**

| Step | Action |
|---|---|
| 1 | Agent execution completes (`agentStop` event) |
| 2 | Hook checks if last action was a Playwright test run |
| 3 | If tests PASSED → do nothing |
| 4 | If tests FAILED → read failure output, identify error |
| 5 | Diagnose root cause (locator mismatch, timeout, assertion, URL, import path) |
| 6 | For locator issues → inspect live page with MCP Playwright, update PageObject |
| 7 | Fix the code in the correct file |
| 8 | Re-run only failed tests using `--last-failed` |
| 9 | If still failing after 2 fix attempts → stop and report to user |

## Login Module

### LoginPage.js (Page Object)

**Path:** `tests/Login/PageObject/LoginPage.js`

```javascript
class LoginPage {
  /**
   * @param {import('@playwright/test').Page} page
   */
  constructor(page) {
    this.page = page;

    // Locators — verified via MCP Playwright snapshot on https://www.saucedemo.com
    this.usernameInput = page.getByRole('textbox', { name: 'Username' });
    this.passwordInput = page.getByRole('textbox', { name: 'Password' });
    this.loginButton = page.getByRole('button', { name: 'Login' });
    this.errorMessage = page.locator('[data-test="error"]');
  }

  /**
   * Navigate to the application URL
   * @param {string} url
   */
  async navigate(url) {
    await this.page.goto(url, { waitUntil: 'domcontentloaded' });
  }

  /**
   * Perform login action (single-step: username + password + Login)
   * @param {string} username
   * @param {string} password
   */
  async login(username, password) {
    await this.usernameInput.fill(username);
    await this.passwordInput.fill(password);
    await this.loginButton.click();
  }

  /**
   * Check if error message is visible on the page
   * @returns {Promise<boolean>}
   */
  async isErrorVisible() {
    return await this.errorMessage.isVisible();
  }

  /**
   * Get the error message text
   * @returns {Promise<string>}
   */
  async getErrorText() {
    return await this.errorMessage.textContent();
  }

  /**
   * Get the page title
   * @returns {Promise<string>}
   */
  async getPageTitle() {
    return await this.page.title();
  }
}

module.exports = LoginPage;
```

**Locators verified on:** https://www.saucedemo.com via MCP Playwright `browser_snapshot`

| Element | Locator Strategy | Locator |
|---|---|---|
| Username input | Role-based | `getByRole('textbox', { name: 'Username' })` |
| Password input | Role-based | `getByRole('textbox', { name: 'Password' })` |
| Login button | Role-based | `getByRole('button', { name: 'Login' })` |
| Error message | CSS data-test | `locator('[data-test="error"]')` |

**Methods:**

| Method | Description |
|---|---|
| `navigate(url)` | Go to the given URL |
| `login(username, password)` | Fill username, password, click Login |
| `isErrorVisible()` | Returns true if error message is visible |
| `getErrorText()` | Returns the error message text |

## Auto-Heal System

The framework has a **two-layer auto-heal system:**

### Layer 1: Playwright Built-in Retry

Configured in `playwright.config.js`:
- **Local:** 1 retry per failed test
- **CI:** 2 retries per failed test

This handles flaky/intermittent failures automatically without any AI intervention.

### Layer 2: Kiro Agent Auto-Heal Hook

Configured in `.kiro/hooks/auto-heal-tests.kiro.hook`:
- **Trigger:** `agentStop` — fires after every Kiro agent execution
- **Action:** If tests failed, the agent automatically:
  1. Reads failure output and identifies the error
  2. Diagnoses root cause (locator mismatch, timeout, assertion, URL, import path)
  3. For locator issues — inspects live page with MCP Playwright
  4. Fixes the correct file (PageObject, Script, or testData)
  5. Re-runs only failed tests using `--last-failed`
  6. Stops after 2 failed fix attempts and reports to user

**Important:** The auto-heal hook works only when tests are run through Kiro agent. For Git Bash terminal runs, paste the failure output into Kiro chat and the agent will fix and re-run.

## Coding Rules Reference

| Rule | Correct | Wrong |
|---|---|---|
| Module system | `const x = require('...')` (CommonJS) | `import x from '...'` (ES modules) |
| Test runner | `@playwright/test` | `jest`, `mocha`, or plain `playwright` |
| Import from Script/ to PageObject/ | `require('../PageObject/LoginPage')` | `require('./PageObject/LoginPage')` |
| Import from Script/ to testData/ | `require('../testData/loginData')` | `require('./testData/loginData')` |
| Cross-module import | `require('../../Login/PageObject/LoginPage')` | `require('./Login/PageObject/LoginPage')` |
| No hardcoded values | All data from `testData/` files | Inline strings in test files |
| Test tagging | `@smoke`, `@regression` in test name | No tags |
| Test IDs | `TC_LOGIN_001`, `TC_SALARY_003` | No IDs |

## Locator Strategy (Priority Order)

```javascript
// 1. Role-based (best — accessible and stable)
page.getByRole('button', { name: 'Login' })
page.getByRole('textbox', { name: 'Username' })

// 2. Placeholder-based
page.getByPlaceholder('Enter your email')

// 3. Label-based
page.getByLabel('Password')

// 4. Test ID
page.getByTestId('submit-btn')

// 5. CSS selector (last resort)
page.locator('#username')
page.locator('.error-message')
```

**Golden rule:** Never guess locators. Always inspect the actual page first using MCP Playwright `browser_navigate` → `browser_snapshot`.

## Allure Report

> Allure provides rich, interactive HTML reports for managers and stakeholders — with charts, test history, categories, and detailed step breakdowns.

### Packages Required

```bash
npm install -D allure-playwright allure-commandline
```

| Package | Purpose |
|---|---|
| `allure-playwright` | Playwright reporter — generates Allure-compatible JSON result files during test runs |
| `allure-commandline` | CLI tool — generates and opens the Allure HTML report from result files |

### Configuration in `playwright.config.js`

```javascript
reporter: [
  ['html', { open: 'never' }],                              // Playwright HTML report
  ['json', { outputFile: 'test-results/results.json' }],    // JSON for --last-failed
  ['allure-playwright', { outputFolder: 'allure-results' }], // Allure results
  ['list']                                                    // Console output
],
```

### npm Scripts

```json
"report:allure:generate": "npx allure generate allure-results --clean -o allure-report",
"report:allure:open": "npx allure open allure-report",
"report:allure": "npx allure generate allure-results --clean -o allure-report && npx allure open allure-report"
```

| Script | Description |
|---|---|
| `npm run report:allure:generate` | Generate Allure HTML report from result files |
| `npm run report:allure:open` | Open previously generated Allure report in browser |
| `npm run report:allure` | Generate AND open Allure report in one command |
| `npm run report` | Open Playwright's built-in HTML report |

### How to Use (Step by Step)

```bash
# Step 1: Run your tests (Allure results are generated automatically in allure-results/)
npx playwright test

# Step 2: Generate and open the Allure report
npm run report:allure

# Or do it in two separate steps:
npm run report:allure:generate    # Generate HTML from allure-results/ → allure-report/
npm run report:allure:open        # Open allure-report/ in browser
```

### Windows PowerShell Workaround

```bash
cmd /c "npx playwright test"
cmd /c "npx allure generate allure-results --clean -o allure-report"
cmd /c "npx allure open allure-report"
```

### What the Allure Report Shows

| Section | Description |
|---|---|
| **Overview Dashboard** | Pass/fail/skip percentages with pie charts and trend graphs |
| **Suites View** | Tests grouped by `test.describe()` blocks (Login Module, Salary Calculator Module) |
| **Timeline** | Execution timeline showing when each test ran |
| **Categories** | Failure categories — broken tests, product defects, etc. |
| **Test Details** | Each test with steps, duration, attachments (screenshots, videos, traces) |
| **History** | Test trends over multiple runs (if allure-results are preserved between runs) |

### Folder Structure After Running

```
project-root/
├── allure-results/          ← Raw JSON results (auto-generated during test run)
│   ├── xxx-result.json      ← One file per test execution
│   └── ...
├── allure-report/           ← Generated HTML report (open in browser)
│   ├── index.html           ← Main report page
│   ├── data/                ← Report data files
│   └── ...
```

### Sharing with Managers

The `allure-report/` folder is fully self-contained HTML. You can:
1. **Zip and email** the `allure-report/` folder
2. **Host on a web server** (any static file server works)
3. **Publish to CI/CD** (GitHub Pages, Netlify, etc.)
4. **Open locally** with `npm run report:allure:open`
